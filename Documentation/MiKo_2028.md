# MiKo_2028: Provide more information than just parameter name in documentation

## Cause

Parameter documentation only repeats the parameter name without providing meaningful description.

## Rule description

Documentation must go beyond just listing parameter names.
It should clearly describe what the parameters represent and how they are used.
Simply stating the parameter's name does not provide enough context or information.
Accurate and detailed documentation makes code easier to understand and maintain.

### Rationale behind

Parameter documentation that only repeats the parameter name provides no useful information to developers.
If the documentation is just the parameter name with different formatting, it adds no value and wastes space in IntelliSense.

Writing meaningful parameter documentation provides several important benefits:

- **Actual Information Value**:
  Documentation should provide information that is not immediately obvious from the parameter name alone.
  This includes constraints, valid ranges, expected formats, or how the parameter affects behavior.

- **Better IntelliSense Experience**:
  When developers see parameter documentation in IntelliSense, they should learn something useful.
  Documentation that just repeats the name is clutter that makes IntelliSense less helpful.

- **Reduced Errors**:
  Detailed parameter documentation helps developers pass correct values.
  Information about valid ranges, formats, or special cases prevents common mistakes.

- **Clear Expectations**:
  Good documentation explains what the parameter represents in the context of the method.
  This helps developers understand not just what the parameter is, but what it is used for.

- **Professional Standards**:
  Meaningful documentation demonstrates care and attention to API usability.
  Documentation that just repeats parameter names suggests laziness or lack of attention to quality.

- **Easier Maintenance**:
  When parameters are well-documented, future developers can understand the method without examining the implementation.
  This makes maintenance and enhancement faster and less error-prone.

## How to fix violations

To fix a violation of this rule, rewrite the parameter documentation to provide meaningful information.

Describe what the parameter represents, what values are valid, what format is expected, or how it affects the method's behavior.
Focus on information that is not obvious from the parameter name alone.

## How to suppress violations

```csharp
#pragma warning disable MiKo_2028
#pragma warning restore MiKo_2028
```
