# MiKo_1526: Suffix delegate properties with `Callback`

## Cause

A property of a delegate type has a name that does not end with the `Callback` suffix.

## Rule description

Properties that hold a delegate type act as callbacks.
Their names should reflect that role by ending with the 'Callback' suffix.

### Rationale behind

A delegate property is not just a value — it is something that gets called.
That is a fundamental part of its role in the code, and the name should make that clear.

A name that does not end with `Callback` is a problem in two ways:

1. Names that end with the delegate type itself, such as `Action`, `Delegate`, or `Func`, only repeat type information that is already visible in the declaration.
2. Names that give no indication of the delegate nature at all, such as `GoOnline`, leave readers with no idea that the property holds a callable delegate.

In both cases, the name fails to communicate the purpose of the property.
When a delegate property carries a name that does not end with `Callback`, a reader cannot tell from the name alone whether the property holds a simple value or something that will be invoked.
That uncertainty slows down reading, increases the chance of mistakes, and makes the code harder to maintain.

Using the `Callback` suffix consistently for all delegate properties provides several important benefits:

- **Improved Readability**:
  A name ending with `Callback` immediately signals that the property holds a delegate that will be invoked at a specific point.
  Readers understand the role of the property without having to trace through the code or inspect the type declaration.

- **Consistent Naming**:
  Using the same `Callback` suffix for all delegate properties creates a predictable pattern across the codebase.
  Developers can instantly recognize delegate properties wherever they appear, regardless of what the rest of the name says.

- **Clearer Intent**:
  A name that ends with `Callback` communicates how the delegate is meant to be used, not just what kind of object it stores.
  This makes the design of the code easier to understand and reason about.

- **Reduced Cognitive Load**:
  When delegate properties follow a consistent naming convention, readers do not need to stop and examine the type to understand what the property does.
  The name alone is enough to set the right expectation.

- **Better Maintainability**:
  A property name that clearly reflects the role of the delegate makes it easier to locate and understand the relevant code when a bug needs to be fixed or behavior needs to be changed.

## How to fix violations

To fix a violation of this rule, rename the property so that its name ends with the `Callback` suffix.

A better solution is to convert the property into a method, especially when the delegate is only ever invoked and never reassigned.
At the call site, invoking the stored delegate and calling a method look identical — the delegate value is accessed through the property first, then invoked, but neither step is visible to the reader.
A method makes that intent direct and explicit, removes the indirection of storing a delegate in a property, and makes the code easier to understand at a glance.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1526
#pragma warning restore MiKo_1526
```
