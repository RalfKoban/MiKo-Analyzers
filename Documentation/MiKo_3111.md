# MiKo_3111: Use specific NUnit constraints instead of comparison-based constraints in assertions

## Cause

A test method uses a generic comparison constraint (such as `Is.EqualTo(...)`, `Is.LessThan(0)`, or `Is.GreaterThan(0)`) instead of a specific NUnit constraint (such as `Is.Zero`) in an assertion.

## Rule description

To improve clarity, use a specific NUnit constraint instead of a generic comparison constraint when asserting for well-known values in tests.
This makes the code easier to read and understand.

### Rationale behind

Using specialized constraints like `Is.Zero`, `Is.Empty`, `Is.Null`, `Is.True`, `Is.False`, `Is.NaN`, `Is.Negative`, and `Is.Positive` makes test assertions more expressive and readable.
The intent of the assertion becomes immediately clear without requiring mental translation of literals or comparisons.

This rule provides several important benefits:

- **Improved Readability**:
  A specific constraint is more expressive and immediately communicates the intent.
  Developers can read and understand the assertion faster than with a generic comparison.

- **Reduced Cognitive Load**:
  Specialized constraints eliminate the need to mentally process literals and comparison operators.
  The assertion reads more like natural language.

- **Better Code Consistency**:
  Using built-in constraints for common cases creates consistency across the test suite.
  All tests checking for the same well-known value use the same clear pattern.

- **Self-Documenting Code**:
  A specific constraint is self-explanatory and requires no additional context or comments.
  The assertion's purpose is obvious even to developers unfamiliar with the test.

- **Clearer Test Intent**:
  Specialized constraints highlight that a well-known value is a significant value being tested.
  Generic equality or comparison checks do not emphasize the importance of the specific value.

- **Aligns with Framework Design**:
  NUnit provides these constraints specifically for such common cases.
  Using them follows the framework's conventions and best practices.

## How to fix violations

To fix a violation of this rule, replace the generic comparison constraint with the corresponding specific NUnit constraint:

| Generic comparison constraint                           | Specific constraint |
|---------------------------------------------------------|---------------------|
| `Is.EqualTo(0)`                                         | `Is.Zero`           |
| `Is.LessThan(0)`                                        | `Is.Negative`       |
| `Is.GreaterThan(0)`                                     | `Is.Positive`       |
| `Is.EqualTo(double.NaN)` / `Is.EqualTo(Double.NaN)`     | `Is.NaN`            |
| `Is.EqualTo(float.NaN)` / `Is.EqualTo(Single.NaN)`      | `Is.NaN`            |
| `Is.EqualTo(true)`                                      | `Is.True`           |
| `Is.EqualTo(false)`                                     | `Is.False`          |
| `Is.EqualTo(null)`                                      | `Is.Null`           |
| `Is.EqualTo("")`                                        | `Is.Empty`          |
| `Is.EqualTo(string.Empty)` / `Is.EqualTo(String.Empty)` | `Is.Empty`          |
| `Has.Count.EqualTo(0)`                                  | `Is.Empty`          |

Negated assertions are fixed the same way, for example:
- `Is.Not.EqualTo(0)` becomes `Is.Not.Zero`
- `Is.Not.EqualTo(null)` becomes `Is.Not.Null`

## How to suppress violations

```csharp
#pragma warning disable MiKo_3111
#pragma warning restore MiKo_3111
```
