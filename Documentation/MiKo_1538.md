# MiKo_1538: Do not prefix events with 'On'

## Cause

An event name starts with the prefix `On`.

## Rule description

Avoid using `On` as a prefix in event names because it is reserved for event handling methods.

### Rationale behind

The `On` prefix has a well-established meaning in .NET: it indicates a method that raises or responds to an event.
For example, `OnClick` is a method that handles or raises the `Click` event.
Prefixing an event itself with `On` blurs this boundary and makes it harder to tell whether a member is an event or a method.

The .NET Framework consistently names events after what happened or what is happening (e.g., `Click`, `Closed`, `PropertyChanged`), never using `On` as a prefix for events.

Avoiding the `On` prefix on events provides several important benefits:

- **Clearer Distinction**:
  When events do not start with `On`, it is immediately clear which members are events and which are event-raising or event-handling methods.
  This reduces confusion when reading unfamiliar code.

- **Consistent Conventions**:
  The .NET Framework and its base class library never prefix event names with `On`.
  Following this convention makes custom code feel natural and consistent with the platform.

- **Improved Readability**:
  Event names without the `On` prefix are shorter and describe what occurred rather than how the event is handled.
  This makes the intent of the event clearer at a glance.

- **Natural Event Handler Names**:
  When an event is named `Click`, the corresponding handler method can be naturally named `OnClick`.
  If the event itself were named `OnClick`, there would be no obvious name left for the handler method.

- **Better API Usability**:
  Public APIs with clean, prefix-free event names are easier to consume.
  Subscribers can immediately understand what the event represents without having to decode naming patterns.

- **Reduced Cognitive Load**:
  Following a single, well-known naming pattern means less mental effort is needed to understand what each member does.
  Developers familiar with .NET conventions will immediately recognize events and their handlers.

## How to fix violations

To fix a violation of this rule, rename the event to remove the `On` prefix.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1538
#pragma warning restore MiKo_1538
```
