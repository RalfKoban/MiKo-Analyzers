# MiKo_1525: Do not prefix non-boolean properties with boolean indicators

## Cause

A property whose name starts with `Is`, `Can`, `Has`, or `Contains` does not return a `bool` value.

## Rule description

Avoid naming properties that do not return a boolean value with the prefixes 'Is', 'Can', 'Has', or 'Contains'.
These prefixes strongly suggest that the property returns 'true' or 'false', so using them on non-boolean properties is confusing and misleading.
Choose a name that does not lead readers to expect a boolean result when the property returns something else.

### Rationale behind

Naming conventions for boolean members are well established in .NET.
Using a boolean-style prefix on a property that does not return a `bool` creates a false promise.
The name says one thing, but the code does another.
This gap causes confusion and leads to mistakes.

Following this rule provides several important benefits:

- **Improved Readability**:
  A property name that does not carry a misleading boolean prefix is easier to understand at a glance.
  Readers do not have to look up the property type to find out that the name is not telling the truth about what the property returns.

- **Reduced Risk of Bugs**:
  When a name implies a boolean but the property returns something else, it is easy to use the property incorrectly.
  A name that does not make a false promise makes the intended usage clearer and reduces the chance of mistakes.

- **Consistency with .NET Conventions**:
  The .NET naming guidelines reserve question-style prefixes like `Is`, `Can`, `Has`, and `Contains` for members that return a boolean.
  Following this convention makes the code feel familiar to other .NET developers and sets a clear expectation.

- **Clearer Intent**:
  A name without a false boolean signal communicates what the property actually represents.
  This makes the design of the code easier to understand and reason about.

- **Reduced Cognitive Load**:
  When names consistently follow the boolean-prefix convention, readers do not need to hold mental exceptions in mind while reading the code.
  Every property named with one of these prefixes can be trusted to return a boolean, with no surprises.

## How to fix violations

To fix a violation of this rule, rename the property so that its name no longer implies a boolean result.
Choose a name that describes what the property actually represents, without using a boolean-style prefix.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1525
#pragma warning restore MiKo_1525
```
