# MiKo_1530: Follow .NET Framework Design Guidelines for method names of DependencyPropertyChanged callbacks

## Cause

A `PropertyChangedCallback` method that is registered via the WPF property system does not follow the required naming convention.

## Rule description

`DependencyProperty` change callback handlers should be named starting with `On`, followed by the dependency property name, followed by `Changed` to clearly show they handle changes of dependency property values.

### Rationale behind

When a `DependencyProperty` is registered in WPF, a `PropertyChangedCallback` method can be provided as part of the property metadata.
The WPF property system calls this method whenever the value of the dependency property changes.

Such a callback method returns nothing (`void`) and takes a `DependencyObject` and a `DependencyPropertyChangedEventArgs` as parameters.

Consistent naming of `PropertyChangedCallback` methods makes it immediately clear what a method does and which dependency property it is connected to.
Without a clear naming convention, it can be hard to tell whether a method is a change callback at all.
It also becomes harder to find which property the callback belongs to.

Following this convention provides several important benefits:

- **Improved Readability**:
  A method named `OnValueChanged` instantly communicates that it is the change callback for a dependency property named `Value`.
  No additional context or documentation is needed to understand its purpose.

- **Easier Navigation**:
  When all `PropertyChangedCallback` methods follow the same pattern, it is much faster to find and identify them in a class.
  A scan of the method list immediately reveals which methods are change callbacks and for which property.

- **Reduced Risk of Confusion**:
  Without a convention, a method could be named in a way that does not hint at its role.
  This can cause confusion about when and how the method is called, especially for developers who are new to the codebase.

- **Alignment with WPF Conventions**:
  The `On(PropertyName)Changed` pattern is well established throughout the WPF framework for `PropertyChangedCallback` implementations, as seen in names such as `OnIsEnabledChanged`, `OnFontSizeChanged`, and `OnBorderBrushChanged`.
  Following the same pattern keeps the code consistent with the wider WPF ecosystem.

- **Better Maintainability**:
  When the naming is consistent and predictable, it is easier to connect a dependency property registration to its corresponding change callback.
  This reduces the effort needed when modifying or extending existing WPF components.

- **Clearer Intent**:
  The `On...Changed` pattern is already familiar from regular event handler overrides such as `OnPropertyChanged`.
  This makes it easier to understand for developers who are new to WPF dependency properties.

## How to fix violations

To fix a violation of this rule, rename the method so that its name starts with `On` and ends with `Changed`.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1530
#pragma warning restore MiKo_1530
```
