# MiKo_3504: Do not use train wrecks

## Cause

A method or property chain contains too many chained calls in a single expression.

## Rule description

Avoid writing long chains of method or property calls, because they tightly couple your code to the internal structure of multiple objects.
This makes the code fragile and harder to read or test.
Instead, expose clear operations on your objects to keep the code simple and maintainable.

### Rationale behind

A long chain of calls is commonly called a "train wreck".
The term comes from how the code visually resembles a line of connected train cars.

Such chains violate the Law of Demeter, which says "do not talk to strangers".
Code that chains through multiple objects knows too much about their internal structure.

Fluent APIs are a legitimate exception to this rule.
They are intentionally designed for chaining, and each call returns the same or a compatible type rather than reaching into unrelated internal objects.

Avoiding train wrecks provides several important benefits:

- **Reduced Coupling**:
  Each intermediate object in a chain is a dependency.
  A change in any of them can break the entire chain.
  Keeping calls short limits the impact of such changes.

- **Improved Readability**:
  A long chain is harder to read than a single, well-named method call.
  Short, meaningful method names make the intent of the code immediately clear.

- **Easier Testing**:
  Long chains require complex mock or stub hierarchies to test properly.
  Short, focused calls are much easier to isolate and test independently.

- **Better Encapsulation**:
  An object that exposes meaningful operations keeps its implementation details hidden.
  This makes the code easier to change without affecting other parts of the system.

- **Fewer Null Reference Errors**:
  Every intermediate call in a chain is a potential source of a null reference exception.
  Fewer calls means fewer places where the code can unexpectedly fail at runtime.

- **Clearer Responsibility**:
  A long chain often signals that no single object owns the operation.
  Encapsulating it in a dedicated method places the responsibility where it belongs.

## How to fix violations

To fix a violation of this rule, refactor the chain by introducing well-named methods or properties on the relevant types that encapsulate the steps of the chain.
Each method should express a clear intent and hide the internal structure it operates on.

## How to suppress violations

```csharp
#pragma warning disable MiKo_3504
#pragma warning restore MiKo_3504
```
