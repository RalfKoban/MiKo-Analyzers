# MiKo_1528: Do not prefix parameters with `arg`

## Cause

A parameter name starts with `arg`, `args`, `argument`, or `arguments`.

## Rule description

Parameters in .NET are also known as arguments, so adding `arg` to their names is redundant.
Use names that describe the parameter's role.

### Rationale behind

Prefixing parameter names with `arg` or similar terms does not add any useful information.
The fact that a parameter is an argument is already clear from its position in a method signature.

Keeping parameter names meaningful and free of redundant prefixes provides several important benefits:

- **Improved Readability**:
  Parameter names that describe the role of the value are much easier to understand at a glance.
  A name like `customer` immediately communicates what the parameter represents, while `argCustomer` forces the reader to mentally strip away the prefix before understanding the intent.

- **Cleaner Code**:
  Removing unnecessary prefixes keeps names shorter and reduces visual noise.
  The code becomes easier to scan and reason about.

- **Better Alignment with .NET Conventions**:
  The .NET naming guidelines recommend descriptive, role-based names for parameters.
  Following these conventions makes the code feel consistent with the rest of the .NET ecosystem and easier to work with for developers familiar with those guidelines.

- **Reduced Confusion**:
  Prefixes like `arg` or `args` can suggest that a parameter holds a collection of arguments, similar to the `args` parameter in a `Main` method.
  Avoiding such prefixes prevents misleading implications about the nature of the value.

## How to fix violations

To fix a violation of this rule, rename the parameter by removing the `arg`, `args`, `argument`, or `arguments` prefix.
Make sure the remaining name accurately describes the role or content of the parameter.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1528
#pragma warning restore MiKo_1528
```
