# MiKo_2053: Document `ArgumentNullException` only for reference type parameters

## Cause

An `ArgumentNullException` is documented for a value-type parameter.

## Rule description

An `ArgumentNullException` should not be documented for value-type parameters as value types inherently cannot have a null value.

### Rationale behind

Value types in .NET, such as int, bool, DateTime, and structs, cannot be null unless they are nullable value types (using the `Nullable<T>` or `T?` syntax).
Documenting an `ArgumentNullException` for a regular value-type parameter is incorrect because such parameters cannot be null, and documenting impossible conditions creates confusion.

Avoiding `ArgumentNullException` documentation for value-type parameters provides several important benefits:

- **Accurate Documentation**:
  Documentation should reflect the actual behavior of the code.
  Value-type parameters cannot be null, so documenting `ArgumentNullException` for them is misleading.

- **Reduced Confusion**:
  When developers see `ArgumentNullException` documented for a parameter, they expect that passing null is possible.
  Incorrect documentation creates confusion about the method's behavior.

- **Better Code Understanding**:
  Correct exception documentation helps developers understand which parameters can be null and which cannot.
  This improves the overall understanding of the API.

- **Proper Type System Understanding**:
  Correct documentation reinforces understanding of the .NET type system and the difference between value types and reference types.
  This helps developers write better code.

- **Cleaner Documentation**:
  Removing incorrect exception documentation makes the overall documentation cleaner and more focused on actual error conditions.
  This improves documentation quality.

## How to fix violations

To fix a violation of this rule, remove the `ArgumentNullException` documentation from value-type parameters.
If the parameter is a nullable value type (T?), the exception documentation may be appropriate, but for regular value types, it should be removed.

## How to suppress violations

```csharp
#pragma warning disable MiKo_2053
#pragma warning restore MiKo_2053
```
