# MiKo_1527: Avoid 'Sub' suffix on type names

## Cause

A type name ends with the `Sub` suffix.

## Rule description

Avoid naming types with the `Sub` suffix.
This suffix does not add useful information because any type can be a supporting type for another type.
Use meaningful, specific type names that make the purpose of the type obvious.

### Rationale behind

Using `Sub` as a suffix is a form of notation akin to Hungarian notation.
It attempts to indicate that a type is a "sub-type" or helper of another type.
However, any type can support or be related to any other type.
Therefore, the suffix conveys no reliable semantic or design information, adds noise, and reduces readability.

Avoiding such suffixes and using clean, intention-revealing names instead provides several important benefits:

- **Improved Readability**:
  Type names without artificial suffixes are easier to read and understand at a glance.
  A well-chosen name immediately communicates what the type represents, without requiring the reader to mentally strip away a meaningless suffix first.

- **Better Maintainability**:
  When type names clearly express their purpose, it is easier to locate relevant code when a bug needs to be fixed or a feature needs to be changed.
  Names that carry no real meaning make it harder to navigate a codebase over time.

- **Reduced Cognitive Load**:
  Suffixes like `Sub` force the reader to process extra noise before understanding what the type actually represents.
  Removing such suffixes means less mental effort is needed to read and understand the code.

- **Consistency with Established Conventions**:
  The .NET naming guidelines do not use Hungarian-style suffixes for type names.
  Following these conventions makes the code feel familiar to other .NET developers and easier to work with.

- **Clearer Intent**:
  A type name should describe what the type represents, not how it relates to other types in the hierarchy.
  Relationships between types change as code evolves, so embedding them in names creates names that quickly become misleading.

## How to fix violations

To fix a violation of this rule, rename the type by removing the `Sub` suffix and choosing a name that clearly describes what the type represents.

## How to suppress violations

```csharp
#pragma warning disable MiKo_1527
#pragma warning restore MiKo_1527
```
